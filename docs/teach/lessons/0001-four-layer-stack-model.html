Game Stack Literacy · Lesson 1

The four-layer stack model

Why “changing the stack is hard” is true at exactly one layer — and how to tell which layer a decision touches.

Every time a stack question comes up — “should this be Next.js? Godot? do I need a server?” — it feels like one big, scary, irreversible choice. It isn’t. A technology stack is four stacked layers, and they have wildly different costs to change. Once you can name the layer a decision lives in, the anxiety mostly evaporates: most changes turn out to be cheap and reversible, and you can see exactly which one is not.

This lesson teaches only the model and one key insight. We ground every layer in this repository, so it stays concrete.

The four layers

▸ Read the stack bottom-up: Layer 1 is the ground everything runs on; Layer 4 is how it reaches a player. Each higher layer wraps the one below without replacing it.

1
Language / runtime — what code is written in and what executes it This project: JavaScript, run by the browser’s JS engine.
Cost to change: HIGH — this is the real “rewrite”
2
Framework — how code is structured inside the runtime This project: none. The UI is hand-written DOM in js/ui.js (694 lines).
Cost to change: MODERATE, contained
3
Build tooling — turns source into shippable files This project: none. index.html loads 16 raw <script> tags by hand.
Cost to change: LOW, additive
4
Packaging / distribution — how it reaches players This project: a browser URL served by python3 -m http.server.
Cost to change: LOW, doesn’t touch logic

Notice what the “This project” lines say: at three of the four layers, this game currently runs on nothing — no framework, no build tool, no packaging beyond a URL. That is not a gap to feel bad about. It is the starting point ADR 0016 deliberately chose, because every one of those empty layers can be filled later, cheaply, when a real need appears.

The one insight that dissolves the dread

Key insight The fear that “stack changes are hard” is almost entirely a Layer 1 phenomenon. Layers 2–4 are incremental and reversible. Everything this project might realistically do — add a build tool, move the UI to a framework, wrap it for Steam — lives in Layers 2–4 and never leaves JavaScript.

Changing Layer 1 means rewriting your code in a different language / runtime — JavaScript → a game engine like Godot (GDScript) or Unity (C#). That is the expensive, genuinely risky move, because your whole codebase is rebuilt. Every other change wraps or restructures the JavaScript you already have:

▸ This is why ADR 0016 says the only Layer-1 change “on the table” is a native engine — and the spec doesn’t require one. The decision isn’t “web vs. engine forever”; it’s “don’t pay the Layer-1 cost speculatively.”

  • L3 Adopt Vite → a build step appears; your .js files are unchanged.
  • L2 Move js/ui.js to Svelte → the UI layer is restructured; combat.js, ai.js, actions.js stay as they are.
  • L4 Wrap in Tauri/Electron for Steam → a native shell appears around the same web build; zero logic rewritten.
  • L1 Rebuild in Godot → everything is rewritten. The only true rewrite in the list.

Why this project already makes Layer 2 cheap

A Layer-2 change (swapping in a UI framework) is only cheap if your game logic isn’t tangled into your UI. Open two files side by side:

js/combat.js — pure logic, no DOM, tested in Node

static computeAttackForce(attacker, targetHex, opts) {
  const base = attacker.calculateMilitary() * (1 + attacker.getAttackBonus());
  const projected = base * CombatSystem.crossingPenalty(targetHex.terrain, ...);
  ...
}

js/ui.js — the DOM layer (694 lines of getElementById / createElement)

combat.js never touches the screen. It takes numbers, returns numbers, and its tests run in plain Node with no browser. ui.js is the opposite — it exists only to push data onto the page. That separation is the whole reason a future framework swap is a L2 move and not a rewrite: a framework would replace ui.js and leave every logic module untouched. ADR 0016 calls preserving this separation “the single discipline that keeps future migration cheap.”

Check yourself

For each change, pick the layer it lives in. Same four options every time — the model is small on purpose. Retrieval is the point; guess, then read the feedback.

“I want to add Vite so I stop hand-ordering <script> tags.”

Vite turns source into shippable files — that is Layer 3. It is additive: your JavaScript and your logic are unchanged.

“I want to rewrite js/ui.js as Svelte components.”

A framework changes how code is structured inside the runtime — Layer 2. It replaces the UI layer only; combat.js and the other logic modules stay put.

“I want to ship this on Steam using a Tauri or Electron wrapper.”

A native shell around the same web build is Layer 4 — packaging/distribution. It does not touch game logic. This is exactly how a web game reaches Steam without an engine.

“I want to rebuild the whole game in Godot.”

This is the one true rewrite — Layer 1. You leave JavaScript for GDScript/C#, and the entire codebase is rebuilt. Justified only by a genre shift to real-time/physics, or by success funding a port — never paid speculatively.
